- (2026-04-18) Built backend bulk-categorize endpoint + audit for issue #22, per Niobe's spec v1.1 (§15 / A-1..A-4, AC-17/18/21/24).
  - **Almost-lost-a-method edit mistake:** Used `edit` to insert a new service method right before `async def add_note(...)`, by replacing a block of text that *ended* with the `add_note` signature line. My `new_str` didn't re-include that signature line, so `add_note`'s body was left orphaned (hung parameter list, no `def`). The rest of the file still parsed (method body became stray statements at class scope!), but `add_note` disappeared from the class. Caught it because I habitually grep for the surrounding method names right after a large insert. **Lesson: when inserting a new method adjacent to an existing one, grep `async def (old|new)` immediately after and confirm both show up at the expected line numbers. Never trust that a multi-line edit left the neighbour intact.**
  - **Router path ordering matters:** Registered `POST /api/transactions/bulk-categorize` *before* `PATCH /api/transactions/{transaction_id}/categorize` in the router module. FastAPI routes are matched in registration order — putting a literal path after a path-param-bearing path means the param swallows the literal. Trivial in isolation; easy to miss when doing a `git diff`-style review.
  - **Audit-field absence is meaningful:** Made `batch_correlation_id` an optional kwarg on `AuditService.log` and *omitted* the `batchCorrelationId` field entirely when the caller doesn't pass it, rather than storing `None`. Keeps single-row audit entries free of misleading empty fields and lets downstream queries filter on `IS_DEFINED(c.batchCorrelationId)`. Added a test asserting both presence and absence.
  - **Per-row vs request-level errors — where to draw the line:** Spec §15 / A-2 makes this call explicit, and I followed it: request-level errors (unknown/inactive category, schema validation, batch-too-large) are HTTP 4xx with a single error body; per-row errors (NOT_FOUND, SPLIT_PARENT_NOT_BULK_UPDATABLE, CONCURRENCY_CONFLICT) are HTTP 200 with a `failed[]` list. The service method raises `ValueError` for the former and returns `(batch_id, succeeded, failed)` for the latter — clean separation.
  - **Per-row resilience:** Wrapped only `repo.replace` in a `try/except Exception`, not the whole per-row block. A bug in my own code (e.g., a KeyError on `existing.get`) should still surface as a 500, not be silently bucketed as `CONCURRENCY_CONFLICT`. Keeping the catch narrow kept the fault model honest.
  - **Split-parent handling is data-driven in v1 but named for Phase 3:** Used the existing `isSplit` bool to detect parents. When Phase 3's `splits[]` model lands, the check will need to update in lockstep — but the stable error code `SPLIT_PARENT_NOT_BULK_UPDATABLE` (AC-24) stays the same, so the frontend/dialog strings don't need to churn.
  - **Pydantic `max_length` on a list = free HTTP 422.** Let the schema enforce the 200-item cap instead of writing a manual guard. The error body is less friendly than a custom `BATCH_TOO_LARGE` string, but it satisfies spec §15 / A-3 ("Backend enforces independently: over-limit returns HTTP 422") with zero lines of business logic.
  - **Flake8 vs black config drift:** `api/pyproject.toml` has black-style `line-length=120` (from CI: `.github/workflows/pr-checks.yml`), but there's no `.flake8` / `setup.cfg` overriding the 79-char default. Running flake8 locally without the CI flags floods you with E501s that CI never sees. Always replicate the CI flags (`--max-line-length=120`) when linting locally in this repo.
