- (2026-04-18) Frontend for bulk-categorize (issue #22), following Niobe's spec v1.1.
  - **Don't fight `displayedColumns` — make it a computed.** The admin/viewer split for the bulk-select column is cleanest as `readonly displayedColumns = computed(() => this.authService.isAdmin() ? ['select', ...base] : base)`. This meant touching three other spots in the template that previously used `displayedColumns` as an array: `[attr.colspan]="displayedColumns().length"`, `*matHeaderRowDef="displayedColumns(); sticky: true"`, and `*matRowDef="let row; columns: displayedColumns()"`. Missing any one breaks MatTable at runtime without a TS error. Grep for every occurrence when converting a property to a getter/signal.
  - **Snapshot the selection when the dialog opens, not when Apply fires.** The spec's EC-7 (filter changes with dialog open) and AC-4 (month-walk preserves selection) both push you toward capturing a frozen `Transaction[]` at `dialog.open()` time and passing it via `MAT_DIALOG_DATA`. If you instead read the live signal inside the dialog, a background filter change quietly mutates the target set under the user's feet. Dialogs are lightweight — pass the full list, not just ids, so the dialog can compute its own net/uncategorized counts without round-tripping to the parent.
  - **Use `allTransactions` for the snapshot, not `transactions()`.** The filtered-visible `transactions()` signal is the *rendered* subset. Month-walk keeps selected rows in `allTransactions` but they may not be visible if the user applied a client-side date trim. Snapshotting from `transactions()` would silently drop them at Apply time. Went with `this.allTransactions.filter(t => ids.has(t.id))` — matches the mental model that "selected" means "across all loaded data."
  - **AC-25 ("succeeded deselected, failed stay selected") changes how you write the success path.** A naive "bulk succeeded → `clearSelection()`" call would also clear the rows that failed, forcing the admin to re-pick them. The correct pattern: always compute the diff (`succeeded.forEach(id => selected.delete(id))`) regardless of whether `failed.length` is zero. In the all-succeeded case this is identical to a clear; in the partial case it's exactly what AC-25 requires. One code path, no branching.
  - **Snackbar with a clickable action for partial failure.** Material's `MatSnackBarRef` exposes `onAction()` which fires when the user taps the action button. That's the right hook for "Tap for details → open results dialog" per spec §8.3 — no custom snackbar component needed.
  - **Stable error-code union with `| string` escape hatch.** The backend may add error codes over time (spec A-2 explicitly reserves that right). Typed the union as the five known codes *plus* `string` and wrote the UI reason-lookup with a `default:` branch that surfaces `f.message` verbatim when the code is unknown. Keeps the UI forward-compatible without requiring a frontend release every time Morpheus adds a code.
  - **i18n parity via the `AppLabels` type is already enforced at build time.** Because `EN_LABELS` and `ES_LABELS` are both declared `: AppLabels`, a missing translation in either file is a TypeScript error. The `labels-parity.spec.ts` file I added is a belt-and-suspenders runtime assertion for when Jest finally gets wired into CI (`jest.config.ts` is staged, but `pr-checks.yml` doesn't invoke it yet). Cheap to keep, and it documents the NB-4 invariant.
  - **Production `ng build` fails in this sandbox because Angular inlines Google Fonts and `fonts.googleapis.com` is blocked.** Dev build works. This isn't a regression from my changes; the repro exists on a fresh checkout too. Don't waste time trying to make it pass locally — CI has internet. I verified dev build + lint both green as the local signal.
